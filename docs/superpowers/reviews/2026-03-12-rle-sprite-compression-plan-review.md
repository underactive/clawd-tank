## Issue 1 (BLOCKING) — Task 6: Wrong PNG source paths for sprite regeneration

**Where:** Task 6, Step 1 and Step 2.

The plan instructs:
```
Run: ls assets/svg-animations/ to find the frame directories.
```
and then runs `png2rgb565.py` against:
```
assets/svg-animations/idle
assets/svg-animations/alert
...
```

`assets/svg-animations/` contains only `.svg` source files (e.g., `clawd-idle-living.svg`). There are no PNG subdirectories there.

The actual PNG frame directories live at:
```
tools/sprite-designer/exports/idle/
tools/sprite-designer/exports/alert/
tools/sprite-designer/exports/happy/
tools/sprite-designer/exports/sleeping/
tools/sprite-designer/exports/disconnected/
```

**However**, those directories hold the original key-frame PNGs (8 frames for idle, 6 for the rest), not the full-rate frame sequences that produced the existing headers (96 / 40 / 20 / 36 / 36 frames). The full sequences were generated by `tools/svg2frames.py` from the SVGs, using Playwright and headless Chromium.

**Required fix:** Task 6 is missing a prerequisite step to regenerate the intermediate PNG frame sequences via `svg2frames.py`. The corrected Step 1 and Step 2 should be:

```bash
# Step 1a: generate PNG frame sequences from SVGs (requires playwright + chromium)
python3 tools/svg2frames.py assets/svg-animations/clawd-idle-living.svg  /tmp/clawd-frames/idle/       --fps 6  --scale 6
python3 tools/svg2frames.py assets/svg-animations/clawd-notification.svg /tmp/clawd-frames/alert/      --fps 10 --scale 6
python3 tools/svg2frames.py assets/svg-animations/clawd-happy.svg        /tmp/clawd-frames/happy/      --fps 10 --scale 6
python3 tools/svg2frames.py assets/svg-animations/clawd-sleeping.svg     /tmp/clawd-frames/sleeping/   --fps 6  --scale 6
python3 tools/svg2frames.py assets/svg-animations/clawd-disconnected.svg /tmp/clawd-frames/disconnected/ --fps 6 --scale 6

# Step 2: regenerate RLE headers (same commands as plan, corrected paths)
python3 tools/png2rgb565.py /tmp/clawd-frames/idle/        firmware/main/assets/sprite_idle.h        --name idle
python3 tools/png2rgb565.py /tmp/clawd-frames/alert/       firmware/main/assets/sprite_alert.h       --name alert
python3 tools/png2rgb565.py /tmp/clawd-frames/happy/       firmware/main/assets/sprite_happy.h       --name happy
python3 tools/png2rgb565.py /tmp/clawd-frames/sleeping/    firmware/main/assets/sprite_sleeping.h    --name sleeping
python3 tools/png2rgb565.py /tmp/clawd-frames/disconnected/ firmware/main/assets/sprite_disconnected.h --name disconnected
```

The fps and scale values should be tuned to match the existing frame counts if frame-count parity with the current headers matters.

## Issue 2 (NON-BLOCKING, MISLEADING) — Task 5 Step 8: inaccurate comment about `apply_sprite_frame` occurrences

**Where:** Task 5, Step 8.

The plan states:
> "(This appears in two places in the function: the normal frame advance and the oneshot-finished branch.)"

This is wrong. `apply_sprite_frame` appears **once** inside `scene_tick` (at line 463 in the current source), not twice. The oneshot-finished branch at line 459 calls `scene_set_clawd_anim()` and then `return`s — it never reaches line 463, and `scene_set_clawd_anim` will itself call `decode_and_apply_frame` after the refactor. The implementer instruction that follows (only replace the one line at 463) is correct; the parenthetical explanation is just wrong and could cause confusion.

**Required fix:** Replace the parenthetical with:
> "(There is one call at line 463, in the normal frame-advance branch. The oneshot-finished branch at line 459 calls `scene_set_clawd_anim()` and returns early; after the refactor, `scene_set_clawd_anim` already calls `decode_and_apply_frame` internally.)"

## Issue 3 (NON-BLOCKING, MINOR) — Task 4 Step 3: placeholder path in encoder validation command

**Where:** Task 4, Step 3.

The validation command reads:
```bash
cd /path/to/clawd-tank
```

`/path/to/clawd-tank` is a placeholder that will cause the command to fail. An implementer running this verbatim from the repo root does not need `cd` at all — `sys.path.insert(0, 'tools')` in the inline Python script already handles discovery. The `cd` line should either be removed or replaced with the actual project path.

**Required fix:** Remove the `cd /path/to/clawd-tank` line, or note that the command must be run from the repository root.

## Everything Else: Verified Correct

The following were checked and are sound:

- **Flash config (Task 1):** `CONFIG_ESPTOOLPY_FLASHSIZE="8MB"` is the correct sdkconfig key. The proposed `partitions.csv` arithmetic is correct: `0x10000 + 0x7F0000 = 0x800000` exactly fills 8 MB with no overflow.
- **`rle_sprite.h` decoder (Task 3):** Both `rle_decode_rgb565` and `rle_decode_argb8888` are correct. BGRA byte layout (`[B, G, R, A]`) matches LVGL's ARGB8888 little-endian format and matches the existing `build_frame_dscs` implementation. The `uint16_t` count field supports runs up to 65535, consistent with the encoder cap.
- **Unit tests (Task 2):** All five test cases are logically correct. The frame-offset test values `{0, 2, 6}` are accurate for the described RLE data. The byte-order assertions in `test_decode_to_argb8888` are consistent with the decoder. The Makefile rule pattern matches `test_config_store`. The `-I../main` flag in `CFLAGS` will resolve `rle_sprite.h` correctly; no new stubs are needed.
- **Python encoder (Task 4):** `rle_encode` correctly caps run length at 65535. `format_rle_array` produces valid C initializers (trailing commas are legal in C99+). The new `generate_header` correctly computes word offsets (`len(runs) * 2`). The `rle_sprite_t` struct and the generated symbol names (`{prefix}_rle_data`, `{prefix}_frame_offsets`) are consistent between the encoder output and the `anim_defs` references in `scene.c`.
- **`scene.c` refactor (Task 5):** The new `anim_def_t` fields (`rle_data`, `frame_offsets`) correctly replace `frames`. The `ensure_frame_buf` / `decode_and_apply_frame` pair correctly mirrors the `free_frame_bufs` / `build_frame_dscs` / `apply_sprite_frame` trio. `TRANSPARENT_KEY` (defined at line 29 of `scene.c`) remains in scope. `malloc()` usage for `frame_buf` is consistent with the existing pattern (PSRAM-backed via `CONFIG_SPIRAM_USE_MALLOC=y`). The width/height fallback (`? def->width : SPRITE_W`) can be safely dropped since all five `anim_defs` set explicit non-zero dimensions.
- **Simulator compatibility:** The simulator's `CMakeLists.txt` includes `${FIRMWARE_DIR}` (i.e., `firmware/main`) in its include path, so the new `#include "rle_sprite.h"` in `scene.c` will resolve identically in the simulator build without any shim changes.
- **File paths for new/modified files:** All paths except the sprite source paths in Task 6 are correct (`firmware/main/rle_sprite.h`, `firmware/test/test_rle_decode.c`, `firmware/sdkconfig.defaults`, `firmware/partitions.csv`).
- **Raw size estimate:** Plan states 13.3 MB; actual calculation is 14.1 MB (96+40+20+36+36 frames × 180×180×2 bytes). Minor inaccuracy in the preamble, does not affect any code.
